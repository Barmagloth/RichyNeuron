## 0. Роль агента и дисциплина

- Агент — **builder**. Реализует ровно то, что в спеке, минимальным диффом.
- Любой результат (код/лог/график) сопровождается версией файла в имени.
- Перед сабмитом — прогнать чеклист из §8.
- Запрещено: менять задачи, метрики, бейзлайны, гиперпараметры бейзлайна
  «чтобы было лучше». Спек заморожен.

## 1. Среда (жёсткое ограничение)

- **PyTorch, CPU-only.** Не предполагать наличие CUDA. Код обязан работать на CPU.
- Если `torch.cuda.is_available()` — использовать, но это НЕ требование и НЕ
  тестируется. Дефолт и CI — CPU.
- Python ≥ 3.10. Зависимости минимальны: `torch`, `numpy`. Графики — `matplotlib`
  (опционально, не в hot path). Никаких triton/causal-conv1d/mamba-ssm.
- Все размеры моделей и задач подобраны так, чтобы один прогон обучения
  укладывался в **минуты на CPU** (см. §3, §5).

## 2. Что реализуется: RichUnit (связь 2 из обсуждения)

Один слой `RichUnitLayer(d_model, d_state)`, обрабатывающий вход
`x: [B, T, d_model]` → выход `[B, T, d_model]`.

Поюнитная рекуррентность по времени (референсная реализация — цикл по t,
T мало). Уравнения на шаг t:

    s_t = alpha * s_{t-1} + (1 - alpha) * (x_t @ W_s)      # [B, d_state]
    g   = sigmoid(x_t @ W_g + s_t @ W_h)                   # [B, d_model]
    v   = x_t @ W_v                                         # [B, d_model]
    y_t = v * g                                             # поэлементно

- `alpha`: обучаемый вектор `[d_state]`, параметризован через
  `alpha = sigmoid(alpha_raw)` чтобы гарантировать (0,1). Инициализация
  `alpha_raw` ~ так, чтобы стартовая alpha ∈ [0.5, 0.95] (разные строки —
  разные тайм-скейлы; инициализировать спредом, не константой).
- Матрицы: `W_s:[d_model,d_state]`, `W_g:[d_model,d_model]`,
  `W_h:[d_state,d_model]`, `W_v:[d_model,d_model]`. Bias у g и v — да.
- Начальное состояние `s_0 = 0`.
- Реализация рекуррентности — **наивный цикл по t** (`for t in range(T)`),
  собирающий выходы в список → `torch.stack`. Корректность важнее скорости.
  НЕ оптимизировать scan на этом этапе.

## 3. Задачи (синтетические, маленькие)

Обе задачи требуют И памяти (temporal), И нелинейной обработки (channel) —
именно там слияние должно давать профит.

### 3.1 Selective Copy
- Алфавит: `n_tokens=8` (+ blank=0, +marker). Длина `T=64`.
- В случайных позициях разбросаны `k=4` «значимых» токена, остальное — blank.
- После marker модель должна выдать значимые токены **в порядке появления**.
- Вход — one-hot или embedding `[B,T,d_model]`. Выход — классификация по позициям.
- Генератор данных — детерминированный по seed. Train/val split по seed,
  не по сэмплам (бесконечный генератор, фиксированные val-seeds).

### 3.2 Associative Recall
- `n_pairs` ключ-значение пар (key,value) подаются последовательно, затем
  query = один из ключей; цель — выдать соответствующее value.
- `n_tokens=16`, `n_pairs=8`, T выводится из этого.
- Тот же контракт генератора (seeded, детерминированный).

Обе задачи — кросс-энтропия. Метрика качества — **token-level accuracy** на
фиксированном val-наборе.

## 4. Бейзлайны (ОБЯЗАТЕЛЬНЫ — без них проект бессмыслен)

Сравниваем RichUnit против разнесённой архитектуры, делающей то же самое
двумя слоями:

- **B1 (stacked):** `[TemporalMixer] → [ChannelMixer]`, где
  - TemporalMixer — минимальный линейный рекуррент (та же scan-форма
    `h_t = a*h_{t-1} + b*x_t`, БЕЗ гейта, БЕЗ связи с channel-частью),
    выход линейный.
  - ChannelMixer — SwiGLU: `(x W1) * silu(x W2) @ W3`. Без состояния.
  - Это «честный стек»: temporal и channel разделены, не видят друг друга
    поюнитно — то, что RichUnit объединяет.
- **B2 (GRU-cell ref):** один `nn.GRU` слой как ходовой sanity-бейзлайн.

Все модели оборачиваются в одинаковый каркас: `embed → [core] → linear head`.
Различается только `[core]`.

## 5. Протокол эксперимента — попытка опровергнуть H0

**Нулевая гипотеза — дефолт, и она НЕ в нашу пользу:**
> H0: RichUnit НЕ достигает целевого качества при меньшем числе параметров, чем
> разнесённый стек B1, с заранее заданным запасом `DELTA_MIN`.

Эксперимент существует, чтобы ПОПЫТАТЬСЯ опровергнуть H0. H1 (выигрыш по
параметрам) считается поддержанной ТОЛЬКО при опровержении H0 по правилу §5.4.
Пока не опровергнута — H0 в силе. Это не симметрично: «не удалось показать
провал» ≠ «показан успех».

### 5.0 Предрегистрация (ОБЯЗАТЕЛЬНО заполнить ДО первого прогона sweep)

Записать значения в `report_*.py` как константы (или в `PREREG.md`) и закоммитить
ДО запуска. Менять после старта запрещено; вынужденное изменение — отдельной
записью «отклонение + причина».

- `ACC_TARGET` — целевое token-acc (предложение: 0.95; см. оговорку §5.1).
- `DELTA_MIN` — минимальное относительное сокращение параметров, считаемое
  значимым. **Предложение: 0.20 (20%).** Выигрыш RichUnit меньше `DELTA_MIN` =
  H0 НЕ опровергнута, даже если число формально меньше.
- `N_SEEDS` — число seed на точку (минимум 3; предложение 5).
- Правило агрегации: точка засчитывается достигшей `ACC_TARGET`, только если
  **median по seed ≥ ACC_TARGET** (не max, не «хотя бы один seed»).
- Бюджет: `max_steps`, оптимизатор (AdamW), lr-свип — одинаковые для ВСЕХ моделей.

### 5.1 Калибровка ACC_TARGET (анти-подгонка)
`ACC_TARGET` калибруется ОДИН раз, по бейзлайну B1, ДО оценки RichUnit: берётся
качество, которого B1 устойчиво достигает на средней ширине сетки. Это
предотвращает выбор таргета, удобного для RichUnit. После калибровки — заморозить.

### 5.2 Подбор минимальной ширины
Для каждой модели (RichUnit, B1, B2) найти наименьшую ширину, при которой
median-по-seed val-качество ≥ `ACC_TARGET` в пределах фиксированного бюджета.
- Сетка `d_model ∈ {16,32,48,64,96,128}` (расширить вверх, если никто не достиг).
- `d_state ∈ {4,8,16}` для моделей с состоянием.
- Бюджет и оптимизатор — из §5.0, идентичны для всех.

### 5.3 Фиксация
Для минимальной достигшей таргета конфигурации каждой модели записать:
#параметров, #слоёв, шагов до таргета, median/min/max accuracy по seed.

### 5.4 Решающее правило (без интерпретационной свободы)
- **H0 опровергнута на задаче**, если параметры RichUnit ≤ (1 − `DELTA_MIN`) ×
  параметры B1 при выполнении обоими `ACC_TARGET` по median-правилу.
- **Поддержка H1 (полная):** H0 опровергнута на ОБЕИХ задачах.
- **Частичная:** ровно на одной. Трактуется как «слабый/неустойчивый сигнал,
  требует репликации», НЕ как успех.
- **H0 НЕ опровергнута (включая выигрыш < DELTA_MIN):** отрицательный результат.
  Это полноценный, обязательный к публикации исход, НЕ провал проекта.

Запрещено: менять `DELTA_MIN`/`ACC_TARGET`/правило агрегации после взгляда на
результаты; выбирать «лучший» seed; объявлять успехом не-целевую метрику (напр.
«зато лучше обучается») — это новая гипотеза для отдельной проверки, не зачёт
текущей (см. RESEARCH_MAP §0, запрет HARKing).

## 6. Контроль самообмана (обязательные ablation/логи)

Отрицательный результат ЛЮБОГО из A1/A2 **аннулирует** заявленный профит
RichUnit, даже если §5.4 формально показала выигрыш: выигрыш без работающей
фичи — артефакт лишних параметров, а не подтверждение гипотезы.

- **A1 — состояние реально используется?** Прогнать выигравшую конфигурацию
  RichUnit с занулённым и замороженным путём состояния (`W_h = 0`). Если
  accuracy НЕ падает заметно — состояние не используется, «богатство» фиктивно,
  профит аннулируется. Записать дельту accuracy обязательно.
- **A2 — коллапс alpha?** Логировать распределение обученных `alpha`. Если масса
  ушла к ~0 или ~1 (нет промежуточных тайм-скейлов) — временно́й элемент
  выродился; флажить, профит под сомнением.
- **A3 — честность бейзлайна.** Подтвердить, что B1/B2 обучались тем же бюджетом,
  оптимизатором и lr-свипом. Недотренированный бейзлайн = недействительный
  результат. Приложить кривые обучения B1 как доказательство сходимости.

## 9. Definition of Done

Проект v0.2.0 завершён, когда получен **вердикт по §5.4 с числами, агрегацией по
seed и ablation A1–A3**, независимо от знака результата. «H0 не опровергнута, вот
цифры» — полноценный успех прототипа: он честно ответил на поставленный вопрос.
Провалом считается только нечестный или неинтерпретируемый результат (подгонка
порогов, отброшенные seed, недотренированный бейзлайн, профит без работающего
ablation).
